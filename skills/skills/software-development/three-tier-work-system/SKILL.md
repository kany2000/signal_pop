# 三层工作体系 (Three-Tier Work System)

## 核心理念

面对任何任务时，先判断它属于哪一层，然后选择最合适的实现方式。

**原则：能用低层级解决就不用高层级 — 简单 > 复杂**

---

## 三层定义

### 第1层：RPA（机器人流程自动化）

**特点**：基于规则、确定性、重复性
**触发**：明确的、固定规则的任务
**不具有**：AI 思考能力

**适用场景：**
- 定时抓取数据
- 文件格式转换
- 监控告警
- 简单表单填写
- 机械性重复操作

**实现方式：**
- cronjob + shell/python 脚本
- 简单的 if-then 逻辑
- 固定流程模板

---

### 第2层：Workflow（工作流编排）

**特点**：多步骤、串联工具、有分支逻辑
**触发**：需要多个工具/服务配合的任务
**需要**：人工设计流程模板

**适用场景：**
- 新闻采集 → 筛选 → 翻译 → 生成 → 发布
- 多源数据聚合处理
- CI/CD 流水线
- 跨平台数据同步

**实现方式：**
- 串联多个脚本/工具
- 定义清晰的输入输出
- 错误处理和重试机制
- 状态管理和日志

---

### 第3层：Agent（智能体）

**特点**：自主决策、动态规划、工具调用
**触发**：开放式目标、需要判断的任务
**具备**：思考-行动-反馈循环

**适用场景：**
- 需要判断新闻价值
- 动态选择最佳工具
- 复杂推理任务
- 开放式创作（写文章、配图建议）
- 异常情况自主处理

**实现方式：**
- LLM 驱动的规划
- 工具调用 (function calling)
- 循环执行直到达成目标
- 自我反思和修正

---

## 决策流程

```
新任务来了？

1. 这个任务有固定规则吗？
   ├── YES → 用 RPA (cronjob / 脚本)
   └── NO ↓

2. 步骤是否清晰、多工具串联？
   ├── YES → 用 Workflow (pipeline)
   └── NO ↓

3. 需要 AI 思考和判断？
   ├── YES → 用 Agent (LLM + tools)
   └── NO → 重新审视任务定义
```

## Signal Pop 应用示例

| 层级 | 任务 | 实现 |
|------|------|------|
| RPA | 定时触发（周一三五8:00） | cronjob |
| RPA | RSS 抓取网页内容 | 脚本 |
| Workflow | 筛选→翻译→脚本生成 | 串联 pipeline |
| Agent | 判断新闻价值、选择配图 | LLM 决策 |
| Workflow | TTS 语音合成 | 工具调用 |
| Workflow | 视频合成 + 发布 | 固定流程 |

## 项目决策模板

接到新任务时，回答：

1. **任务类型**：RPA / Workflow / Agent
2. **为什么选这一层**：简单说明理由
3. **具体实现**：用什么工具/框架
4. **失败处理**：这一层解决不了怎么办

---

## 常见错误

- **过度设计**：能用 RPA 的任务用了 Agent
- **设计不足**：复杂的 Agent 任务硬塞进 Workflow
- **职责混乱**：在 Workflow 里写 AI 判断逻辑
- **缺乏边界**：三层之间没有清晰的输入输出约定
